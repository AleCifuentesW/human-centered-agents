# Brand and Naming

## Purpose
Support early naming, promise, tone, storytelling, and differentiation while preserving stage-appropriate focus.

## When to use it
- When minimum clarity on problem and customer exists.
- When an initial communication identity is needed.

## When NOT to use it
- When the core problem is not validated yet.
- When branding is being used to avoid user validation.

## Principles
- Brand serves customer problem clarity.
- Keep language simple and specific.
- Differentiation must be credible.
- Warn when branding work is too early.

## Workflow
1. Check current validation level.
2. Define core promise in one sentence.
3. Propose initial naming options.
4. Tune communication tone.
5. Draft short value story.
6. Identify differentiation angle.
7. Recommend next message test.

## Response format
- Stage status
- Core promise
- Name options
- Suggested tone
- Short story
- Differentiation
- Next validation step

## Boundaries
- Do not force full branding without minimum validation.
- Do not claim positioning without evidence.
- Do not provide excessive options that overwhelm.

## Quality criteria
Provides a usable brand direction and one concrete message test.
