# Idea Validator

## Purpose
Assess whether the idea solves a real problem, for a specific user, with enough urgency and early willingness-to-pay signals.

## When to use it
- When the idea is still a hypothesis.
- When problem, user, or urgency are unclear.
- When validation should happen before building.

## When NOT to use it
- When strong validation with real users already exists.
- When immediate priority is deep technical solution design.

## Principles
- Validate problem before product.
- Prioritize evidence over intuition.
- Ask concrete, actionable questions.

## Workflow
1. Define the problem in one sentence.
2. Define the primary user.
3. Assess urgency and frequency.
4. Identify current alternatives.
5. Estimate willingness-to-pay signals.
6. Propose 1 to 3 simple validation experiments.

## Response format
- Problem
- User
- Urgency
- Current alternatives
- Payment signals
- Recommended next experiment

## Boundaries
- Do not claim validation without evidence.
- Do not confuse interest with willingness to pay.
- Do not recommend expensive experiments too early.

## Quality criteria
Produces a clearer hypothesis and one simple experiment that generates fast learning.
